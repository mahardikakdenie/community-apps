# NexGate

Welcome to the NexGate repository! NexGate (formerly Udin Park) is a community-driven application built using Expo and React Native, designed to provide seamless location-based services, real-time presence features (like check-ins), and a rich interactive map experience using MapLibre.

## 📂 Folder Structure

Understanding the project's folder structure is crucial for effective collaboration. Below is an overview of the key directories and their purposes:

- **`app/`**: Contains the core application screens and routing logic using Expo Router. This is where file-based routing happens.
  - **`app/(tabs)/`**: Contains the main tab navigation screens (e.g., Map, Cafe).
- **`assets/`**: Stores static assets such as images, icons, and fonts used throughout the application.
- **`components/`**: Reusable UI components (e.g., `MapComponent.tsx`, `Sidebar.tsx`) that can be shared across different screens to maintain a consistent design system.
- **`constants/`**: Holds application-wide constants, configuration values, and styling tokens (like colors and typography).
- **`hooks/`**: Custom React hooks for encapsulating complex state management and side effects.
- **`scripts/`**: Utility scripts for build processes, maintenance, or code generation.
- **`android/`**: Native Android project files generated by Expo Prebuild (for native module linking).

## 🚀 Getting Started

Follow these steps to set up the project locally on your machine.

### Prerequisites

- [Node.js](https://nodejs.org/) (LTS version recommended)
- [npm](https://www.npmjs.com/) or [yarn](https://yarnpkg.com/)
- [Expo CLI](https://docs.expo.dev/get-started/installation/)

### Installation

1. **Clone the repository**
   ```bash
   git clone https://github.com/mahardikakdenie/community-apps.git
   cd community-apps
   ```

2. **Install dependencies**
   ```bash
   npm install
   ```

3. **Start the development server**
   ```bash
   npx expo start
   ```

## 🤝 How to Collaborate

We welcome contributions from the community! To ensure a smooth collaboration process, please follow these guidelines:

### 1. Branching Strategy
- **`master`**: Represents the stable production-ready code.
- Create a new branch for every feature or bug fix. Use descriptive names like `feature/add-user-profile` or `bugfix/map-rendering-issue`.

### 2. Making Changes
- Ensure your code follows the existing style and conventions.
- Test your changes thoroughly on the target platforms.
- If you add new components or screens, please document them or include comments for clarity.

### 3. Submitting a Pull Request (PR)
- Push your branch to the remote repository.
- Open a Pull Request against the `master` branch.
- Provide a clear and detailed description of your changes, including screenshots or videos if you altered the UI.
- Wait for a code review and address any feedback provided by the maintainers.
